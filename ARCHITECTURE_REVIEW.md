# Serkonia — аудит текущей архитектуры и план перехода к сервисному подходу

## 1) Краткая оценка текущего состояния

Текущая база проекта **хорошая для MVP на GitHub Pages**:
- простая структура файлов;
- данные вынесены в JSON;
- UI уже разделён на основные зоны (topbar / sidebar / content / info-panel);
- базовые сценарии (переключения режимов, загрузка данных, рендер) работают.

Для стадии «один автор + быстрые итерации» это релевантно и оправдано.

## 2) Что уже сделано правильно

1. **Разделение слоёв по типам файлов**: HTML / CSS / JS / JSON.
2. **Data-driven подход**: маркеры и таймлайн читаются из `data/*.json`.
3. **Понятный UI-каркас** для дальнейшего расширения.
4. **Работа через GitHub Pages** — быстрый деплой без серверной инфраструктуры.

## 3) Ограничения текущей реализации

### 3.1 JS как единый монолит
Весь runtime-код находится в `js/app.js`, и там одновременно:
- состояние приложения;
- загрузка данных;
- рендеринг карты;
- рендеринг таймлайна;
- управление панелью;
- обработчики drag/click.

Это нормально для прототипа, но будет мешать при добавлении:
- фильтров по эпохам/фракциям;
- связи событий таймлайна с маркерами карты;
- мультиязычности;
- редактора контента;
- сохранения пользовательских настроек.

### 3.2 Сильная связность DOM и логики
Код напрямую привязан к конкретным DOM-id. Любое изменение верстки повышает риск регрессий.

### 3.3 Нет схем и валидации данных
`markers.json` и `timeline.json` не проходят формальную проверку. Со временем это часто приводит к «тихим» поломкам при ручном редактировании контента.

### 3.4 Безопасность рендера таймлайна
В `renderTimeline()` используется `innerHTML` для вставки полей из JSON. Если позже появится внешний источник данных, это создаст XSS-риск.

### 3.5 Масштабирование данных
При росте количества маркеров/событий текущий подход без индексов/виртуализации начнёт деградировать по UX.

## 4) Рекомендованная целевая архитектура (поэтапно, без «переписать всё»)

## Этап A (без backend, остаёмся на GitHub Pages)

Цель: перейти к **модульной фронтовой архитектуре**.

### Предлагаемое разделение папок

```text
index.html
css/
  style.css
js/
  app.js                  # точка входа/инициализация
  core/
    state.js              # единый store состояния
    events.js             # event-bus/pub-sub
    config.js             # константы/фичефлаги
  services/
    data.service.js       # загрузка JSON + кэш
    marker.service.js     # операции по маркерам
    timeline.service.js   # операции по событиям
    storage.service.js    # localStorage (режимы, фильтры)
  ui/
    switches.js
    sidebar.js
    map.js
    timeline.js
    info-panel.js
  utils/
    dom.js
    sanitize.js
    validators.js
data/
  markers.json
  timeline.json
  schemas/
    markers.schema.json
    timeline.schema.json
```

### Что это даст
- меньше связности между блоками;
- проще тестировать по частям;
- проще подключать будущие функции как отдельные сервисы.

## Этап B (контент как сервис)

Цель: отделить «движок сайта» от «контента мира».

### Варианты
1. **Оставить JSON в репозитории**, но ввести:
   - versioning (`meta.version`);
   - минимальный CI-check JSON-схем.
2. **Headless CMS / BaaS** (например, Supabase/Firebase/Directus) для редактирования без PR.

Рекомендация: начать с варианта 1, чтобы не усложнять инфраструктуру преждевременно.

## Этап C (микросервисы — когда реально понадобится)

Полноценные микросервисы нужны, когда появляются:
- авторизация;
- роли и совместная редактура;
- пользовательские заметки/закладки;
- медиа-хранилище;
- аналитика и поисковые индексы.

До этого момента лучше использовать **модульный фронт + 1 backend API** (или serverless functions) вместо раннего дробления на много сервисов.

## 5) Предлагаемое разделение на «сервисы» уже сейчас

Даже без сервера можно ввести сервисный слой внутри фронта:

1. **DataService**
   - `loadMarkers()`
   - `loadTimeline()`
   - `warmCache()`
2. **MapService**
   - нормализация маркеров;
   - фильтрация по группам/эпохам/фракциям.
3. **TimelineService**
   - сортировка/группировка событий;
   - связь с картой через `relatedMarkerIds`.
4. **PanelService**
   - формирование ViewModel для info-panel.
5. **PreferencesService**
   - хранение режима карты, открытости панели, активных фильтров в `localStorage`.

Это уже «сервисный подход», но без operational overhead микросервисов.

## 6) Приоритетные оптимизации (в порядке пользы)

1. **Разбить `app.js` на модули** (минимум: state/data/ui).
2. **Убрать `innerHTML` в таймлайне** в пользу `textContent` + `createElement`.
3. **Ввести JSON Schema и проверку в CI**.
4. **Синхронизировать карту и таймлайн** через общую модель ссылок.
5. **Добавить типизацию через JSDoc или TypeScript (по желанию)**.
6. **Добавить unit-тесты на сервисы и валидацию данных**.

## 7) Минимальный roadmap на 4 итерации

### Итерация 1 (технический фундамент)
- модульный JS;
- DataService + Store;
- без изменения дизайна.

### Итерация 2 (качество данных)
- schema для markers/timeline;
- линтер/валидатор JSON в GitHub Actions.

### Итерация 3 (функционал мира)
- связь событий и точек карты;
- фильтры (эпоха, тип, фракция);
- сохранение фильтров локально.

### Итерация 4 (подготовка к backend)
- адаптер доступа к данным (local JSON ↔ API);
- черновой контракт REST/GraphQL под будущий редактор.

## 8) Итоговая рекомендация

Для Serkonia оптимальный путь:
1. **Не уходить сразу в микросервисы**.
2. Сначала сделать **сервисную модульность внутри фронта**.
3. Затем перейти к **одному API/Serverless-слою**, когда появятся функции совместной работы или авторизации.

Такой путь минимизирует риски, сохраняет скорость разработки и оставляет проект расширяемым под будущие функции DnD-мира.
