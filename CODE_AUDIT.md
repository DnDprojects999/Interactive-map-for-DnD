# CODE AUDIT — Serkonia (2026-04-08)

## Scope
- `js/app.js`
- `js/modules/*.js`
- Runtime patterns around rendering, event handling and data flow.

## Executive summary
Проект уже в хорошем состоянии для MVP: есть модульность (`data`, `ui`, `map`, `editor`, `state`), понятная инициализация и data-driven рендер.

Перед "глубоким режимом редактирования" логично сначала укрепить базовые гарантии качества. Ниже — рекомендации с объяснением **зачем**, **что менять**, и **какой эффект на код**.

---

## Recommendation 1 — Валидация данных перед рендером (Priority: High)

### Почему
Сейчас `loadData()` проверяет только `response.ok`, но не структуру JSON. Если в `markers.json` опечатка в ключах или неожиданный тип поля, ошибка проявится уже в UI и не всегда очевидно где.

### Что менять
1. Добавить JSON Schema для `markers`, `timeline`, `archive`.
2. Проверять схемы в CI (pre-merge).
3. (Опционально) В runtime делать мягкую валидацию с понятными ошибками.

### Влияние на код
- Появится явный контракт между контентом и кодом.
- Станет проще расширять модели (новые поля/режимы), потому что нарушение контракта будет видно сразу.
- Снизится количество "тихих" регрессий.

---

## Recommendation 2 — Контролируемые изменения state (Priority: Medium)

### Почему
Мутации `state` распределены по нескольким модулям. Это не критично для текущего размера, но при росте функционала сложнее понимать источник побочных эффектов.

### Что менять
1. Перейти к action-подходу (`applyAction`/`dispatch`) постепенно.
2. Либо минимум: ввести централизованные set-функции для ключевых частей состояния.

### Влияние на код
- Предсказуемее поведение при переключении режимов.
- Легче дебажить и покрывать тестами отдельные сценарии.
- Ниже вероятность "скрытых" поломок в связанных модулях.

---

## Recommendation 3 — Скрытая функция редактора (Priority: Medium)

### Почему
Горячая клавиша edit-mode должна оставаться служебной (только для редактора), а не пользовательской публичной функцией.

### Что менять
Оставить включение edit-mode только при явном editor-access:
- `?editor=1` в URL, **или**
- `localStorage['serkonia:editor-access'] = 'granted'`.

### Влияние на код
- Обычный пользователь не активирует редактор случайно.
- Редактор получает контролируемый "скрытый" вход без визуального шума в UI.
- Риск случайного изменения контента заметно ниже.

---

## Recommendation 4 — Проверки модульных зависимостей (Priority: High)

### Почему
Ты верно заметил: без явных проверок ошибок при инициализации модулей сложнее быстро понять, что сломалось (DOM, API модуля, отсутствие метода и т.д.).

### Что менять
1. Валидировать наличие обязательных DOM-узлов на старте (`getElements`).
2. Проверять публичный API модулей (`ui`, `mapModule`, `editor`) сразу после создания.
3. Бросать ранние и понятные ошибки с названием модуля/метода.

### Влияние на код
- Fail-fast поведение вместо поздних `undefined is not a function`.
- Быстрее локализуются причины проблем после изменений в разметке или модулях.
- Упрощается сопровождение при ближайших итерациях изменений.
